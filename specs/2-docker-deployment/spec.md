# 简化 OIDC Provider 的 Docker 部署

## 1. 概要（Feature Summary）

本特性旨在为现有的简化 OIDC Provider 提供一套标准化的 Docker 部署能力，使开发者和运维人员可以通过容器镜像在本地、测试和基础生产环境中一致地运行该服务。

目标是补全当前规格中缺失的“部署层面”约定，围绕镜像构建、配置注入和运行方式形成稳定契约，而不改变既有的 OIDC 协议行为。

---

## 2. 业务问题与动机（Problem & Motivation）

当前状态：

- 授权码流的协议行为、端点契约和测试体系已经稳定，并有完整的 spec/plan/tasks 文档。
- 开发和测试主要通过本地 Python 环境与 CLI 启动服务（例如 `python -m app.cli dev`）。
- 缺少标准化的部署技术方案，导致：
  - 各环境的依赖和配置容易出现漂移；
  - 搭建新环境成本高（需要重复安装依赖、配置 Python 运行时）；
  - 无法快速用容器方式在 CI 或其他机器上拉起一个可运行的 OIDC Provider。

引入 Docker 部署能力的动机：

- 为开发者提供“一条命令拉起服务”的体验，降低环境准备成本。
- 为 CI、预发布环境提供可复用的标准镜像，提升测试与验证的一致性。
- 为后续扩展（如引入反向代理、日志收集、监控等）提供统一的容器化基础。

---

## 3. 目标（Goals）

1. 提供一个可重复构建的 Docker 镜像，用于运行当前的 OIDC Provider 应用。
2. 支持使用环境变量配置核心运行参数（如 `DATABASE_URL`、`OIDC_ISSUER_URL`、端口等），避免将敏感信息写死在镜像中。
3. 提供面向开发与测试场景的容器运行指引（例如本地单容器运行、与已有 SQLite 文件协作）。
4. 使 CI 能够基于该镜像拉起服务并运行现有自动化测试（或其子集），验证授权码流行为。
5. 保持与现有快速起步文档（quickstart）在用户体验上的一致性：用户既可以用本地 Python 环境，也可以用 Docker 完成同样的流程。

---

## 4. 非目标（Non-Goals）

本特性阶段明确不包含以下内容：

- 不设计和实现 Kubernetes、Nomad 等编排层方案；
- 不引入多服务拆分或服务网格，仅关注当前单体 OIDC Provider 的容器化；
- 不解决所有生产级别的运维问题（如自动伸缩、蓝绿发布、集中日志与监控系统），仅关注基础部署与运行；
- 不改变现有的 OIDC 协议行为和外部端点契约；
- 不引入额外的数据库或外部依赖（维持当前 SQLite 模式，未来如需引入其他数据库将作为独立特性）。

---

## 5. 角色与用户（Actors & Users）

- **开发者（Developer）**
  - 希望在本地通过 Docker 快速拉起 OIDC Provider，与现有测试和客户端集成。

- **测试工程师 / CI 系统（Tester / CI）**
  - 希望在 CI 流水线中使用统一的容器镜像，运行端到端或集成测试。

- **运维人员 / 内部平台（Ops / Platform）**
  - 希望在内网或测试环境中以 Docker 形式部署该 Provider，作为其他应用的认证源。

---

## 6. 用户场景与验收（User Scenarios & Acceptance）

### 场景 D1：开发者本地使用 Docker 拉起 OIDC Provider

**描述**

开发者无需安装 Python 依赖，仅需安装 Docker，即可在本地通过容器运行 OIDC Provider，并按照授权码流 quickstart 完成登录与 UserInfo 获取。

**步骤**

1. 开发者在项目根目录执行镜像构建命令（或从镜像仓库拉取已发布镜像）。
2. 开发者通过 `docker run` 或 `docker-compose up` 启动容器：
   - 暴露默认端口（例如宿主机的 8000）；
   - 配置必要的环境变量（如 `DATABASE_URL`、`OIDC_ISSUER_URL` 等）。
3. 容器启动后，开发者访问 `/.well-known/openid-configuration`、`/authorize` 等端点，验证服务正常运行。
4. 开发者按现有 quickstart 文档中的步骤，走通授权码 + token + userinfo 完整流程。

**验收要点**

- 使用 Docker 运行时，无需在宿主机额外安装 Python 依赖。
- 通过容器运行的服务，其对外可见的端点和行为与非容器化版本一致。
- quickstart 中的流程（适当替换 base URL）可以在 Docker 部署下完整通过。

---

### 场景 D2：CI 使用 Docker 镜像运行自动化测试

**描述**

CI 流水线使用统一构建的 Docker 镜像拉起 OIDC Provider，执行自动化测试验证授权码流行为。

**步骤**

1. CI 任务拉取或构建最新版本的 OIDC Provider 镜像。
2. CI 在隔离环境中运行容器：
   - 指定测试数据库路径（例如临时 SQLite 文件）；
   - 配置 issuer、端口等环境变量。
3. CI 通过现有测试命令（或其变种）对部署好的服务执行测试：
   - 包括 Discovery、Authorize、Token、UserInfo、JWKS 等端点的集成或 E2E 测试。
4. CI 根据测试结果决定是否通过该构建。

**验收要点**

- 使用同一镜像可以在本地和 CI 环境中复现服务行为。
- 测试命令在容器化场景下可稳定执行且结果可预期。
- 测试失败时，日志可以通过容器标准输出或挂载卷方便地获取。

---

### 场景 D3：内部环境使用 Docker 部署稳定的 OIDC Provider

**描述**

内部环境（如开发或测试集群）使用 Docker 运行 OIDC Provider，为多个客户端应用提供认证服务。

**步骤**

1. 运维人员基于标准镜像，在指定环境中部署容器（可单机或简单集群）。
2. 为容器配置持久化存储（如挂载 SQLite 文件所在目录）以及必要的环境变量。
3. 内部客户端使用该 Provider 的 issuer 和端点完成集成。

**验收要点**

- 部署方式与开发/CI 场景使用同一个镜像和相同的配置模式（环境变量为主）。
- 服务重启后，持久化数据（用户、客户端、授权码历史）符合预期（由运维根据挂载策略决定持久化程度）。

---

## 7. 功能需求（Functional Requirements）

### 7.1 镜像构建与运行

- **FR-D1**：系统必须提供一个可构建的 Docker 镜像定义（例如 Dockerfile），用于打包当前 OIDC Provider 应用及其依赖。
- **FR-D2**：镜像构建过程必须可在无额外手工步骤的环境中完成（例如 CI），只依赖代码仓库内容和标准工具。
- **FR-D3**：运行容器时，必须通过环境变量配置关键运行参数，包括但不限于：
  - `DATABASE_URL`
  - `OIDC_ISSUER_URL`
  - HTTP 监听地址和端口（如 `HOST`、`PORT` 或等效配置）。
- **FR-D4**：容器必须暴露一个可配置的 HTTP 端口（默认与当前应用一致，例如 8000），以供外部访问。

### 7.2 与现有行为的一致性

- **FR-D5**：在 Docker 部署下，以下端点必须保持与现有实现相同的协议行为和路径：
  - `/.well-known/openid-configuration`
  - `/authorize`（GET/POST）
  - `/token`
  - `/userinfo`
  - `/jwks`
- **FR-D6**：在 Docker 部署下，现有自动化测试（或其经过环境变量配置的变体）应能在不修改业务断言的前提下通过。

### 7.3 配置与初始化

- **FR-D7**：必须支持在容器启动前通过环境变量或命令行参数执行数据库初始化和 demo 数据种子（例如复用 `app.cli` 中现有命令）。
- **FR-D8**：必须在文档中明确说明在不同场景下（本地演示 / CI / 内部环境）推荐的配置组合和初始化步骤。

---

## 8. 非功能需求（Non-Functional Requirements）

### 8.1 安全性

- **NFR-D1**：镜像中不得硬编码生产级别的密码、客户端密钥或其他敏感信息，这些内容必须通过环境变量或外部配置注入。
- **NFR-D2**：镜像构建过程不应泄露本地敏感文件（例如开发者机器上的配置或密钥）。

### 8.2 可运维性

- **NFR-D3**：容器内应用的日志必须输出到标准输出/标准错误，以便与常见的日志收集系统集成。
- **NFR-D4**：应用在容器内异常退出时应返回非零退出码，便于编排系统或脚本检测。

### 8.3 可移植性

- **NFR-D5**：镜像应在常见的 Docker 运行环境中正常运行（如本地 Docker Desktop、常见 CI 运行器）。
- **NFR-D6**：镜像构建过程尽量避免与特定宿主操作系统强耦合的行为，以便在不同平台上复现。

---

## 9. 假设与依赖（Assumptions & Dependencies）

- **A-D1**：有可用的 Docker 运行环境（本地或 CI），且由使用方负责安装和维护。
- **A-D2**：当前数据库仍然使用 SQLite，且通过 `DATABASE_URL` 配置；未来如切换到其他数据库，将在新的特性中扩展。
- **A-D3**：对外网络访问、反向代理、TLS 终止等由上游基础设施负责，本特性不直接涉及。

---

## 10. 成功标准（Success Criteria）

- **SC-D1：本地容器化运行可用**  
  开发者只安装 Docker，即可按照文档构建/拉取镜像并通过 quickstart 场景完成授权码登录 + UserInfo 获取。

- **SC-D2：CI 可复用镜像运行测试**  
  CI 能够使用同一镜像拉起 OIDC Provider，并运行现有测试套件（或其明确定义的子集）验证授权码流的关键行为。

- **SC-D3：部署方式对外契约稳定**  
  在 Docker 部署下，外部客户端只需关注 issuer 和端点契约，不需要感知应用内部部署方式。

- **SC-D4：为后续运维能力扩展预留空间**  
  在不破坏现有容器化方案的前提下，可以平滑引入更复杂的部署方式（如编排、监控、日志收集），容器镜像和配置模式仍然适用。
