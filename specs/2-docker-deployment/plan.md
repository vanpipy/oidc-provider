## 1. 技术上下文（Technical Context）

### 1.1 现有应用特性回顾

- 授权码流协议行为已在 `specs/1-oidc-auth-code` 中定义，并在实现与测试中落地。
- 应用当前通过 FastAPI + Uvicorn 运行，开发场景下使用 CLI 命令（如 `python -m app.cli dev`）启动。
- 配置管理依赖环境变量和配置文件（例如数据库 URL、issuer、token 过期时间等）。

### 1.2 容器化相关约束

- 目标是为现有应用增加 Docker 部署能力，而非重写运行时或引入新的 Web 框架。
- 容器运行时将通过环境变量注入与当前实现一致的配置字段。
- 数据持久化在初期仍使用 SQLite，具体文件位置由 `DATABASE_URL` 和挂载策略决定。

---

## 2. 宪章检查（Constitution Check）

### 2.1 关键约束映射

- 分层架构
  - 容器化不应打破现有的接口层 / 应用层 / 领域层 / 基础设施层职责划分。
  - Docker 相关内容主要体现在部署与运行方式，不侵入业务代码。
- 可测试性
  - 通过容器运行的应用应仍可被现有测试套件覆盖。
  - 允许在 CI 中以容器方式运行测试，以验证行为一致性。
- 安全
  - 镜像中不得硬编码密码或客户端密钥。
  - 依旧遵守授权码一次性使用等 OIDC 安全约束。

### 2.2 宪章 Gate 评估

- 分层约束：容器化主要涉及运行方式，满足“业务逻辑与部署解耦”的方向性要求。
- 可测试性：容器化后仍保留当前测试入口，甚至可作为额外验证手段。
- 安全集：镜像构建过程需遵守不泄露敏感信息的约束。

结论：在宪章约束下，本 Docker 部署计划允许向下执行。

## 3. Phase 1：镜像构建方案设计

目标：确定 Docker 镜像的基本结构与构建方式，为后续实现提供统一参考。

### 3.1 基本决策

- 保持现有 Python 运行时与依赖集合，通过镜像封装环境。
- 使用合适的基础镜像（例如某个 Python 官方镜像的小体积变体），减少镜像体积并保持可维护性。
- 使用分层构建策略（如安装依赖与拷贝代码分层），以便缓存重用。

### 3.2 设计要点

- 安装依赖
  - 使用项目已有的依赖定义文件（如 `pyproject.toml`）在镜像内安装依赖。
- 运行入口
  - 复用现有 CLI 逻辑或直接运行应用入口，使镜像在容器启动时自动拉起 HTTP 服务。
- 配置注入
  - 依靠环境变量配置数据库、issuer、端口等字段，与当前应用保持一致。

## 4. Phase 2：实现初版 Docker 镜像

目标：实现满足最小可用要求的 Docker 镜像，使开发者能在本地构建并运行。

### 4.1 构建步骤

- 在仓库中新增 Docker 镜像定义文件。
- 在构建阶段安装项目依赖，并拷贝应用代码。
- 使用适当的容器内工作目录与用户身份运行应用。

### 4.2 本地验证

- 在本地构建镜像并运行容器：
  - 映射端口到宿主机；
  - 配置简单的 `DATABASE_URL` 与 issuer。
- 使用现有 quickstart 中的步骤，验证授权码流与 UserInfo 获取是否正常工作。

---

## 5. Phase 3：与测试和 CI 的集成

目标：使现有测试流程可以选择在容器环境中执行，作为行为一致性的验证手段。

### 5.1 测试集成策略

- 保持 `python -m app.cli test` 作为逻辑入口。
- 在需要时，通过容器运行测试：
  - 将代码目录或镜像中的代码作为测试目标；
  - 通过环境变量确保测试使用合适的数据库与配置。

### 5.2 CI 集成要点

- 在 CI 配置中增加构建镜像的步骤。
- 在需要时拉起容器并执行测试命令，确保与本地行为一致。
